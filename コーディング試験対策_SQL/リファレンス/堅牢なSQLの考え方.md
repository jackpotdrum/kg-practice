# 堅牢なSQLの考え方

コーディング試験で「動くSQL」から一歩進んで、
**何度実行しても同じ結果になりやすく、仕様変更にも耐えやすいSQL**を書くためのメモです。

---

## 1. 先に結論

堅牢なSQLは、次の3つを満たす。

- 再現性: 同じ入力なら毎回同じ結果順・同じ行が返る
- 要件耐性: 仕様の追加・変更に追従しやすい
- 読解性: 後から見ても意図と処理順が分かる

---

## 2. 再現性の担保（最重要）

### なぜ必要か
`ORDER BY total_sales DESC` のように1キーだけだと、同じ値（同点）が出たときの順序は不定になることがある。
その結果、`ROW_NUMBER()` の順位や最終出力順が実行ごとに揺れる可能性がある。

### 実務/試験での基本ルール
- 同点が起こりうる列で並べるときは、**必ずタイブレーク列を追加**する
- 要件に明記されていなくても、安定化のために次キーを入れる
- 最後は一意に近い列（例: `id`）まで入れて「完全決定順序」を作る

### 例
```sql
ORDER BY total_sales DESC, total_qty DESC, product_id ASC
```

この並びは次の意味:
- 第1優先: 売上が高い順
- 同点時: 数量が多い順
- さらに同点時: product_id が小さい順

これで順位計算と最終表示の両方が安定しやすくなる。

---

## 3. よくある「堅牢化ポイント」

### 3-1. 重複排除ルールを明示する
- 例: 同一 `(order_id, product_id)` は `ingested_at` 最新1件を採用
- 同時刻同点に備えて `item_event_id` などでタイブレーク

```sql
ROW_NUMBER() OVER (
  PARTITION BY order_id, product_id
  ORDER BY ingested_at DESC, item_event_id DESC
)
```

### 3-2. 粒度を固定してから集計する
- 1行の意味（例: カテゴリ x 商品）を先に決める
- `GROUP BY` はその粒度に合わせる

### 3-3. 期間条件は「開始以上・終了未満」で書く
- 月末境界や時刻型での取りこぼしを防ぎやすい

```sql
WHERE ordered_at >= '2026-03-01'
  AND ordered_at <  '2026-04-01'
```

### 3-4. 段階分割（CTE）で仕様を分離する
- `base_filter` -> `dedup` -> `agg` -> `ranked` のように分ける
- 1段階1責務にするとレビューしやすい

---

## 4. GROUP BY と PARTITION BY の役割分担

- `GROUP BY`: 行数を減らして集計値を作る
- `PARTITION BY`: 行を残したまま、グループ内順位や累積を計算する

Top-N per group の定番は次の順序:
1. 集計（`GROUP BY`）
2. 順位付け（`ROW_NUMBER() OVER (PARTITION BY ...)`）
3. `rank <= N` で抽出

- [ ] 重複排除のルールを明示したか
- [ ] 集計粒度（`GROUP BY`）は要件と一致しているか
- [ ] `PARTITION BY` は「グループ内」の意味になっているか
